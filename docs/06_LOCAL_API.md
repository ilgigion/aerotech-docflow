# Локальный API

Минимальный FastAPI-сервер предоставляет два endpoint:

- `GET /health` — проверка процесса без обращения к сканеру;
- `POST /scan` — синхронный запуск существующего `document_flow`.

Сервер привязан только к `127.0.0.1:8000`. В нём нет очереди, worker, polling,
Planfix-интеграции, API key/HMAC и внешнего туннеля. Не публикуйте этот процесс
на внешнем интерфейсе без отдельного защищённого шлюза.

## Установка и запуск

```powershell
pip install -r requirements.txt
python -m app.run_local_api
```

Проверка состояния:

```powershell
curl http://127.0.0.1:8000/health
```

Запрос на сканирование:

```powershell
Invoke-RestMethod `
  -Method Post `
  -Uri "http://127.0.0.1:8000/scan" `
  -ContentType "application/json" `
  -Body '{
    "task_id": "53243",
    "doc_type": "НКЛ",
    "document_number": "001",
    "scanner_profile": "EPSON DS-790WN",
    "idempotency_key": "planfix_53243_НКЛ_001"
  }'
```

`idempotency_key` можно не передавать. Тогда сервер стабильно формирует его из
`task_id`, нормализованного типа документа и нормализованного номера.
Повторный успешный запрос с тем же ключом возвращает сохранённый результат с
`scan_executed: false`.

`scanner_profile` обязателен и должен точно совпадать с именем профиля в NAPS2
для учётной записи Windows, под которой работает служба. Значение текущего
запроса безопасно переопределяет только `scanner.profile`; остальные параметры
сканера остаются в `config.toml`. Профиль участвует в fingerprint
идемпотентности: тот же ключ с другим профилем получает HTTP 409.

`document_datetime` больше не входит в HTTP-контракт и считается лишним полем.
Дата и время имени фиксируются на сервере в `Europe/Moscow` непосредственно
перед запуском NAPS2, после получения scanner lock. Формат имени остаётся
`ТИП_ГГММДД_ЧЧММСС_НОМЕР.pdf`.

`task_id`, `doc_type` и `document_number` должны сразу приходить в каноническом
file-safe виде. API не заменяет `/`, `\\`, пробелы и другие неоднозначные символы
молча: такой запрос получает 422. В production `doc_type` дополнительно
проверяется по allowlist, а год серверного времени сканирования — по
разрешённому диапазону лет.

В `scanner_profile` разрешены кириллица, пробелы и обычные символы имени
профиля. Пустая строка, управляющие/скрытые форматирующие символы и значение
длиннее 200 символов отклоняются с 422.

Успешный ответ содержит `file_name`, но намеренно не раскрывает локальный
`file_path`. Занятый сканер и выполняющаяся идемпотентная операция дают HTTP 409,
ошибка входных данных — 422; timeout, недоступный сетевой сканер или
необходимость ручного восстановления — 503; остальные ошибки
сканирования/хранилища — 500.

`operation_id` в HTTP-ответе всегда создаётся до входа в document flow и
передаётся во все стадии. Поэтому при контролируемой ошибке идентификатор в
ответе совпадает с идентификатором в scanner/storage/idempotency-логах.

Настройки NAPS2, входной директории и архива загружаются самим `document_flow` из
существующих переменных окружения и значений по умолчанию проекта.

При `DOCFLOW_ENV=production` безопасных значений по умолчанию нет: сервер до
открытия порта проверяет production-конфигурацию и прекращает запуск при
неподтверждённом, тестовом или недоступном архиве. Та же проверка доступна через
`python -m app.preflight`; она не запускает сканер и не пишет в архив.

В будущем локальный endpoint может вызываться ПланФиксом через отдельный внешний
HTTPS-туннель. Такой туннель и его аутентификация не входят в текущую реализацию.

```powershell
python -m tests.manual.run_scan_epson_profile

```
