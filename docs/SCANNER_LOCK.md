# Защита от одновременного сканирования

## 1. Зачем нужна защита

Сканер — это один физический ресурс. Он не должен получать два задания одновременно.

Без защиты возможны проблемы:

- два процесса NAPS2 одновременно обращаются к одному TWAIN/WIA-сканеру;
- драйвер возвращает `scanner busy`;
- NAPS2 зависает;
- устройство остаётся в странном состоянии;
- оператор нажимает кнопку дважды и получает два запуска;
- временные PDF и архивные имена начинают конфликтовать;
- сложно понять, какое сканирование к какой задаче относится.

Поэтому нужен технический механизм, который гарантирует:

> В один момент времени выполняется только одна операция сканирования.

---

## 2. Что такое lock

Lock — это блокировка общего ресурса.

В нашем случае общий ресурс:

```text
сканер + NAPS2 + временная папка + перенос в архив
```

Когда процесс начинает сканирование, он захватывает lock. Пока lock активен, второй запуск получает понятную ошибку:

```text
Сканер уже используется. Дождитесь завершения текущего сканирования и повторите попытку.
```

После завершения операции lock освобождается.

---

## 3. Выбранное решение сейчас: file lock

Для текущего этапа выбран file lock.

Файл блокировки:

```text
D:\incoming\.scanner.lock
```

Если используется другой `incoming_dir`, lock лежит там:

```text
<incoming_dir>\.scanner.lock
```

Пример содержимого:

```json
{
  "operation_id": "SCAN_20260714_153012_ab12",
  "task_id": "TEST_005",
  "pid": 12345,
  "hostname": "SCAN-SERVER-01",
  "created_at_utc": "2026-07-14T12:30:12.123456+00:00"
}
```

---

## 4. Почему file lock подходит сейчас

Плюсы:

- простая реализация;
- работает между разными Python-процессами;
- не требует БД;
- легко посмотреть состояние руками;
- подходит для одного локального scanner-сервера;
- хорошо сочетается с текущей архитектурой.

Минусы:

- если процесс аварийно завершился, lock-файл может остаться;
- нужно распознавать старый lock;
- на сетевых папках file lock может быть менее предсказуемым;
- это не полноценная очередь задач.

Для текущего проекта это достаточный и практичный уровень защиты.

---

## 5. Почему не обычный `threading.Lock`

`threading.Lock` работает только внутри одного Python-процесса.

Он не защитит, если:

- запущено два экземпляра приложения;
- FastAPI работает в нескольких worker-процессах;
- тест запущен отдельно из терминала;
- один процесс Python завершился, а другой стартовал.

Для нашего случая `threading.Lock` недостаточен.

---

## 6. Почему lock должен быть атомарным

Плохой вариант:

```python
if not lock_file.exists():
    lock_file.write_text("locked")
```

Проблема: два процесса могут одновременно проверить, что файла нет, и оба начнут сканирование.

Правильный вариант:

```python
os.open(lock_path, os.O_CREAT | os.O_EXCL | os.O_WRONLY)
```

Флаг `O_EXCL` означает:

> создать файл только если его ещё нет.

Такой способ защищает от гонки двух процессов.

---

## 7. Как работает текущий `app/locks.py`

Файл:

```text
app/locks.py
```

Основные элементы:

```python
ScannerLockSettings
ScannerLockError
ScannerLockBusyError
ScannerFileLock
scanner_lock()
read_lock_info()
force_remove_lock()
```

Пример использования:

```python
from pathlib import Path
from app.locks import scanner_lock

with scanner_lock(
    lock_path=Path(r"D:\incoming\.scanner.lock"),
    operation_id="SCAN_20260714_153012_ab12",
    task_id="TEST_005",
):
    # scanner.py + storage.py
    ...
```

В `document_flow.py` lock уже подключён вокруг критического участка:

```text
захват lock
→ scan_document()
→ store_document()
→ освобождение lock
```

## 8. Порядок операции

Рекомендуемый порядок:

```text
1. Проверить входные данные.
2. Сформировать ожидаемое имя файла.
3. Захватить lock.
4. Запустить сканирование.
5. Получить временный PDF.
6. Перенести PDF в архив.
7. Освободить lock.
```

Почему сначала проверяются входные данные:

- если `doc_type` пустой, не нужно занимать сканер;
- если `document_number` неправильный, не нужно запускать NAPS2;
- если дата некорректная, ошибка должна появиться до физического сканирования.

---

## 9. Что такое stale lock

Stale lock — это старый lock-файл, который остался после аварии.

Пример:

```text
lock создан
Python аварийно завершился
lock-файл остался
новый запуск думает, что сканер занят
```

Для этого в lock-файле хранится:

```text
created_at_utc
pid
hostname
```

Текущая логика:

- если lock молодой — считаем, что сканирование ещё идёт;
- если lock старый — проверяем процесс;
- если lock создан на этой же машине и процесс ещё жив — не трогаем;
- если lock старый и процесс не жив — lock можно заменить.

Настройка:

```python
ScannerLockSettings(
    stale_after_seconds=30 * 60
)
```

По умолчанию lock считается старым через 30 минут.

---

## 10. Что делать, если lock остался вручную

Проверить lock:

```powershell
Get-Content "D:\incoming\.scanner.lock"
```

Проверить процессы:

```powershell
Get-Process | Where-Object { $_.ProcessName -like "python*" -or $_.ProcessName -like "NAPS2*" }
```

Если точно нет активного сканирования, удалить lock:

```powershell
Remove-Item "D:\incoming\.scanner.lock"
```

Или через Python:

```python
from pathlib import Path
from app.locks import force_remove_lock

force_remove_lock(Path(r"D:\incoming\.scanner.lock"))
```

Важно: не удалять lock, если NAPS2 или Python ещё сканирует.

## 11. Альтернативы file lock

### 11.1. Windows named mutex

Плюсы:

- надёжно на одном Windows-компьютере;
- автоматически освобождается при завершении процесса;
- не оставляет lock-файл.

Минусы:

- привязка к Windows;
- сложнее отлаживать;
- lock не виден как обычный файл;
- может потребоваться `pywin32`.

Вывод: хороший вариант позже, но file lock проще для текущего этапа.

---

### 11.2. SQLite lock / scan_operations

Идея: хранить операции в таблице:

```text
scan_operations
```

Поля:

```text
operation_id
task_id
status
temp_path
final_path
error_code
created_at
started_at
finished_at
```

Плюсы:

- видна история;
- можно делать retry;
- можно видеть статус;
- можно связать с Planfix.

Минусы:

- нужно проектировать БД;
- нужны транзакции;
- больше кода.

Вывод: хороший следующий уровень после file lock.

---

### 11.3. Очередь + worker

Архитектурно лучший вариант для промышленной версии:

```text
запрос → очередь → один scanner worker → сканер
```

Плюсы:

- сканер всегда обрабатывает задания по одному;
- HTTP-запрос не висит 120 секунд;
- можно показывать статус;
- можно повторять задачи;
- можно интегрировать с Planfix.

Минусы:

- нужен worker;
- нужна очередь;
- нужна БД или внешний брокер;
- больше архитектуры.

Вывод: это правильное направление позже, но не сейчас.

---

### 11.4. Planfix как очередь задач

Planfix можно использовать как бизнес-очередь:

```text
Ожидает сканирования
Сканируется
Отсканировано
Ошибка сканирования
Ошибка сохранения
```

Но Planfix не должен полностью заменять локальный lock.

Правильное разделение:

```text
Planfix queue
  отвечает за бизнес-очередь и статусы

Local scanner service
  выполняет физическое сканирование

Local file lock
  защищает сканер от одновременного доступа
```

Почему Planfix не заменяет lock:

- два backend-процесса могут одновременно взять задачу;
- оператор может нажать кнопку дважды;
- статус в Planfix обновляется не мгновенно;
- может пропасть сеть;
- NAPS2 GUI может держать сканер;
- TWAIN-драйвер может ещё не освободить устройство.

Вывод: Planfix-очередь полезна позже, но локальный lock всё равно нужен.

```text
OK: второй запуск заблокирован

TEST 3: после выхода lock должен удалиться
OK: lock удалён
```

Перед запуском:

- NAPS2 GUI закрыт;

## 13. Текущая рекомендация

На текущем этапе использовать:

```text
file lock на весь процесс document_flow
```

То есть lock держится не только на время NAPS2, а до конца переноса файла в архив.

Причина: так проще и безопаснее. Мы защищаем весь критический участок:

```text
сканирование → временный PDF → перенос в архив
```

Позже, когда будет очередь + worker, lock всё равно можно оставить как последний уровень защиты физического сканера.
