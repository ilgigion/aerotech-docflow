# 3.3. Атомарный перенос PDF в архив

## Задача

Если сканирование прошло успешно, временный PDF появляется в `incoming_dir`.

Дальше `storage.py` должен перенести файл в архив:

```text
D:\incoming\PF_....pdf
↓
D:\archive_test\2026\УПД\УПД_260710_101025_2455B.pdf
```

Простой `shutil.move()` работает для первой версии, но у него есть риск: если перенос оборвётся, в архиве может появиться неполный файл или останется неясное состояние.

Цель 3.3 — сделать перенос безопаснее.

---

## Выбранная схема

```text
1. Проверить исходный PDF.
2. Скопировать его во временный .tmp-файл рядом с будущим архивным файлом.
3. Проверить размер временной копии.
4. Проверить, что временная копия похожа на PDF.
5. Переименовать .tmp в финальное имя.
6. Удалить исходный файл из incoming.
```

Пример:

```text
source:
  D:\incoming\PF_TEST_001.pdf

temp:
  D:\archive_test\2026\УПД\.УПД_260710_101025_2455B.pdf.8f3a1c2b9a11.tmp

final:
  D:\archive_test\2026\УПД\УПД_260710_101025_2455B.pdf
```

---

## Почему .tmp создаётся рядом с финальным файлом

Временный файл создаётся в той же папке, что и финальный PDF.

Это важно, потому что финальный шаг:

```text
.tmp -> .pdf
```

происходит внутри одной папки.

Такой шаг намного безопаснее, чем сразу копировать в финальное имя.

---

## Что происходит при ошибке

### Ошибка до финального переименования

Если ошибка произошла во время копирования или проверки:

```text
source остаётся в incoming
.tmp удаляется
final не появляется
```

Значит можно исправить проблему и повторить сохранение без нового сканирования.

### Ошибка после финального переименования

Если финальный файл уже создан, а удалить source не удалось:

```text
final остаётся в архиве
source может остаться в incoming
операция логирует warning
```

Это не считается критической ошибкой, потому что документ уже сохранён в архив.

---

## Что проверяется

`storage.py` проверяет:

```text
source существует
source является файлом
source похож на PDF
archive_root корректен
папка archive_root / ГОД / ТИП создана
имя файла уникально
.tmp скопирован полностью по размеру
.tmp похож на PDF
```

---

## Важное ограничение

Эта реализация не является полноценной защитой от параллельных workers сама по себе.

Защита от параллельного выполнения сейчас обеспечивается через:

```text
app/locks.py
```

То есть нормальный полный процесс:

```text
scanner lock
→ scan_document()
→ atomic store_document()
→ release lock
```

Если позже появится очередь + несколько workers, нужно будет дополнительно подумать о:

```text
SQLite-транзакциях
exclusive create финального файла
едином worker для физического сканера
```

---

## Почему не просто `shutil.move()`

`shutil.move()` может:

```text
внутри одного диска сделать rename
между дисками сделать copy + delete
```

Если архив находится на другом диске или сетевой папке, фактически это может быть копирование. При сбое можно получить непонятное промежуточное состояние.

Новая схема явно контролирует процесс:

```text
copy to .tmp
verify
atomic final rename
delete source
```
