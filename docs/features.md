Отлично. Сейчас у нас уже есть рабочий базовый конвейер:

```text
scanner.py       → получает временный PDF
naming.py        → формирует имя
storage.py       → переносит в архив
document_flow.py → объединяет процесс
```

Теперь задача — не добавлять новые этапы, а сделать этот блок надёжным. Ниже разбор потенциальных проблем, какие последствия они дают и как их решать. Я разделю по сложности и приоритету.

---

# 1. Главные риски текущей версии

Самые опасные места сейчас такие:

```text
1. Одновременный запуск двух сканирований.
2. Сканирование прошло, но перенос в архив не удался.
3. Файл в архиве может получить имя-дубль.
4. NAPS2 может зависнуть или оставить процесс после ошибки.
5. Автоматическое создание папок может создать архив не там, где ожидали.
6. PDF может быть создан, но быть повреждённым или пустым.
7. Ошибки драйвера/сети/сканера пока распознаются не полностью.
8. Логов пока недостаточно для разбора инцидентов.
```

Текущий код хорош как минимальная версия. Но для рабочего использования его надо постепенно усилить.

# 3. Средние по сложности задачи

Эти задачи уже важны для реальной эксплуатации.

## 3.1. Защита от одновременных сканирований

Сейчас если два человека одновременно запустят сканирование, произойдёт конфликт:

```text
1. Два процесса NAPS2 попытаются занять один сканер.
2. Один процесс может зависнуть.
3. Второй может получить ошибку "scanner busy".
4. Оператор не поймёт, что происходит.
```

Решение: добавить **lock**.

Пока можно сделать простой файловый lock:

```text
D:\incoming\.scanner.lock
```

Логика:

```text
если lock существует:
    вернуть ошибку "Сканер уже используется"
если lock не существует:
    создать lock
    выполнить сканирование
    удалить lock
```

Обязательно удалить lock в `finally`, чтобы он снимался даже при ошибке.

Проблема: если Python аварийно завершится, lock может остаться. Поэтому в lock нужно писать:

```text
operation_id
task_id
created_at
process_id
```

И проверять возраст lock-файла. Например, если lock старше 30 минут — считать его зависшим.

Сложность: **средняя**.

Приоритет: **очень высокий**.

Это первое, что я бы добавил.

## 3.2. Сценарий: сканирование прошло, но архив недоступен

Сейчас возможна ситуация:

```text
scanner.py создал временный PDF
storage.py не смог перенести файл
```

Причины:

```text
архивная папка недоступна
нет прав
нет сети
нет места на диске
файл с таким именем заблокирован
```

Что должно происходить:

```text
1. Временный файл нельзя удалять.
2. Пользователю нужно вернуть ошибку "Скан создан, но не сохранён в архив".
3. В ответе должен быть temp_path.
4. Должна быть возможность повторить только перенос без повторного сканирования.
```

Для этого позже стоит добавить отдельную функцию:

```python
store_existing_scan(
    temp_scan_path,
    doc_type,
    document_datetime,
    document_number,
)
```

Так оператору не придётся заново сканировать документ, если ошибка была только в архиве.

Сложность: **средняя**.

Приоритет: **очень высокий**.

## 3.3. Атомарный перенос файла в архив

Сейчас `storage.py` использует `shutil.move`. Это нормально для первой версии, но в эксплуатации лучше сделать безопаснее.

Проблема:

```text
Если перенос прервётся, в архиве может появиться неполный файл.
```

Более надёжная схема:

```text
1. Копируем PDF во временный файл рядом с финальным:
   УПД_260710_101025_2455B.pdf.tmp

2. Проверяем размер и PDF-заголовок.

3. Переименовываем .tmp в финальное имя.

4. Удаляем временный файл из incoming.
```

Переименование внутри одной папки обычно атомарное. Это лучше, чем сразу писать финальный PDF.

Сложность: **средняя**.

Приоритет: **высокий**.

---

## 3.4. Защита от дублей при параллельной работе

Сейчас у нас есть логика:

```text
если файл существует:
    добавить _01
    потом _02
```

Но есть тонкий риск: два процесса одновременно могут проверить, что файла нет, и оба выбрать одно имя.

Решение:

```text
1. На время сохранения использовать общий lock архива.
2. Или создавать файл в режиме exclusive create.
3. Или весь процесс сканирования делать строго однопоточным.
```

Для текущего проекта проще всего:

```text
один общий lock на весь процесс сканирования и сохранения
```

Тогда два процесса не смогут одновременно выбрать одно и то же имя.

Сложность: **средняя**.

Приоритет: **высокий**.

---

## 3.5. Контроль временной папки

Со временем в `D:\incoming` могут остаться файлы:

```text
после ошибок
после ручных тестов
после падения процесса
после отменённого сканирования
```

Нужна политика:

```text
1. Наши файлы называются PF_...
2. Ручные файлы не трогаем.
3. Старые PF-файлы старше N дней можно переносить в quarantine или удалять.
```

Лучше не удалять сразу, а сделать папку:

```text
D:\incoming\_failed
```

И переносить туда проблемные файлы.

Сложность: **средняя**.

Приоритет: **средний**.

# 4. Сложные задачи

Это уже ближе к промышленной эксплуатации.

## 4.1. Очередь заданий

Сейчас процесс синхронный:

```text

```

Для локального теста это нормально.

Но в реальной системе лучше:

```text
1. Запрос создаёт scan_job.
2. scan_job попадает в очередь.
3. Отдельный worker выполняет сканирование.

```

Плюсы:

```text

можно повторять задачи
можно ограничить один scanner worker
```

Минусы:

```text

нужны статусы
нужно больше кода
```

Сложность: **сложная**.

Приоритет: **позже**.

Пока не делаем.

## 4.2. Идемпотентность

Идемпотентность означает: если один и тот же запрос отправили два раза, система не должна создать два разных скана случайно.

Пример проблемы:

```text
оператор нажал "Сканировать"
страница зависла
оператор нажал ещё раз
получили два одинаковых PDF
```

Решение:

```text
task_id + operation_id
```

И хранить в БД:

```text
task_id
status
temp_path
final_path
created_at
finished_at
error
```

Сложность: **сложная**.

Приоритет: **позже**, когда подключим FastAPI/Planfix.

## storage.py

Возможные проблемы:

```text
архивный диск недоступен
нет прав на запись
нет места
архивный путь ошибочно указан
файл с таким именем уже есть
перенос прервался
файл заблокирован
путь слишком длинный
```

Решения:

```text
проверка свободного места
проверка прав записи
атомарный перенос через .tmp
защита от дублей
ограничение длины путей
не создавать archive_root случайно в неправильном месте
```

Важный момент: сейчас мы включили автоматическое создание папок. Это удобно, но есть риск:

```text
ARCHIVE_ROOT указан ошибочно как D:\archve_test
```

Код создаст новую папку `D:\archve_test`, и оператор может не заметить, что файлы уходят не туда.

Решение для боевого режима:

```text
archive_root должен существовать заранее
год/тип можно создавать автоматически
```

То есть:

```text
D:\archive_test             — должен существовать
D:\archive_test\2026        — можно создать
D:\archive_test\2026\УПД    — можно создать
```

## document_flow.py

Возможные проблемы:

```text
скан прошёл, сохранение не прошло
ошибка имени после сканирования
ошибка архива после сканирования
временный файл остался
нельзя понять, на каком этапе ошибка
```

Решения:

```text
проверять naming до запуска scanner.py
сначала сформировать финальное имя
только потом запускать сканер
после ошибки storage возвращать temp_path
добавить stage в ошибки
```

Очень важное улучшение: сейчас лучше изменить порядок.

Текущий порядок:

```text
1. сканировать
2. формировать имя
3. переносить
```

Лучше:

```text
1. проверить входные данные
2. сформировать имя
3. проверить архив
4. сканировать
5. переносить
```

Почему лучше: если оператор передал плохой номер документа, мы узнаем это **до сканирования**, а не после. Тогда не будет ситуации, когда скан уже сделан, а имя сформировать нельзя.

# 7. Что я предлагаю сделать первым

Первое улучшение — **lock на сканирование**.

Причина: это самая практическая защита. Один сканер физически не должен получать два задания одновременно.

Сделаем отдельный файл:

```text
app/locks.py
```

Он будет давать конструкцию:

```python
with scanner_lock(...):
    # здесь scanner.py + storage.py
```

И тогда весь процесс станет безопаснее:

```text
если один скан уже идёт:
    второй запрос не запускает NAPS2
    оператор сразу получает понятное сообщение
```

После lock я бы сразу изменил `document_flow.py`, чтобы порядок был такой:

```text
1. проверить имя
2. проверить/подготовить архив
3. взять lock
4. сканировать
5. перенести
6. вернуть результат
```

Это даст хороший фундамент для дальнейших проверок.
