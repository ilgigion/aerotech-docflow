# 5. Ежедневная работа

## Роли

Оператор выполняет только штатный запуск, проверку `/health`, подготовку
документа и отправку запроса. Администратор меняет конфиг, выполняет recovery,
обновляет приложение и работает с marker/idempotency.

## Начало рабочего дня

1. Убедитесь, что выбран правильный сканер и сеть.
2. Если VPN меняет маршрут к сетевому сканеру, отключите его до сканирования.
3. Закройте NAPS2 GUI, чтобы он не занимал устройство.
4. Проверьте, что в `D:\incoming` нет неожиданного активного lock.
5. Запустите сервер.

Ручной запуск:

```powershell
& "C:\Program Files\Aerotech Docflow\start-manually.ps1"
```

Окно с сервером должно оставаться открытым. В другом терминале:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/health
```

Успешный ответ:

```text
status      : ok
service     : aerotech-docflow
scanner_api : local
version     : 1.0.0-rc1+b45d4c5f
```

`/health` подтверждает только работу API и конфигурации запуска. Он не
проверяет бумагу, доступность сканера или успешность будущего PDF.

## Подготовка документа

Перед каждым запросом:

- удалите скобы и посторонние предметы;
- выровняйте листы;
- положите только один логический документ;
- проверьте ориентацию;
- убедитесь, что автоподатчик не пуст;
- подготовьте уникальные `task_id` и `idempotency_key`.

## Отправка запроса

```powershell
$body = @{
    task_id = "53243"
    doc_type = "НКЛ"
    document_number = "001"
    idempotency_key = "planfix_53243_НКЛ_001"
} | ConvertTo-Json

$response = Invoke-RestMethod `
  -Method Post `
  -Uri "http://127.0.0.1:8000/scan" `
  -ContentType "application/json" `
  -Body $body

$response | Format-List
```

Успешный ответ:

```text
status          : succeeded
file_name       : НКЛ_260721_135000_001.pdf
operation_id    : SCAN_20260721_135102_a1b2c3
task_id         : 53243
idempotency_key : planfix_53243_НКЛ_001
scan_executed   : True
```

`scan_executed: True` означает, что этот запрос физически запускал NAPS2.
`False` означает безопасный повтор ранее успешного результата.

## После успешного запроса

Оператор проверяет:

1. HTTP-статус успешный;
2. `status` равен `succeeded`;
3. `file_name` соответствует реквизитам;
4. PDF существует в `D:\Archive\ГОД\ТИП`;
5. PDF открывается и содержит нужные страницы;
6. в журнале есть тот же `operation_id` и `task_id`.

## Повтор запроса

Если клиент не получил ответ из-за своего timeout, не создавайте новый ключ.
Повторите тот же JSON с тем же `idempotency_key`. Возможны результаты:

- `scan_executed: false` — документ уже сохранён, нового скана не было;
- HTTP 409 — предыдущая операция ещё выполняется;
- HTTP 503 `manual_recovery_required` — нужно проверить состояние;
- controlled 500 — storage/scanner завершились ошибкой, следуйте диагностике.

Новый `idempotency_key` означает новую бизнес-операцию и может запустить новый
физический скан.

## Два одновременных запроса

Физический сканер один. Первый запрос захватывает `.scanner.lock`. Второй запрос
не должен запускать NAPS2 параллельно и получает HTTP 409. После завершения
первого запрос можно повторить.

Не пытайтесь обходить 409 удалением lock во время работающего NAPS2.

## Ошибка во время скана

Не нажимайте запрос много раз подряд. Сохраните:

- HTTP-код;
- тело JSON;
- `operation_id`;
- `error_code`;
- время;
- состояние бумаги и сканера.

Затем выполните read-only диагностику из
[раздела восстановления](09_DIAGNOSTICS_AND_RECOVERY.md).

## Окончание рабочего дня

При ручном режиме нажмите `Ctrl+C` в окне сервера. Дождитесь возврата приглашения
PowerShell. Проверьте отсутствие активного процесса:

```powershell
Get-Process aerotech-docflow -ErrorAction SilentlyContinue
```

Если команда ничего не вернула, сервер остановлен. Недоступность `/health` после
этого нормальна.

## Смена рабочего места или сканера

Профили NAPS2 принадлежат Windows-пользователю. Дома и в офисе могут быть разные
профили. Перед сменой:

1. остановите сервер;
2. проверьте нужный профиль в NAPS2 GUI;
3. измените только `scanner.profile`;
4. выполните `show-config` и `preflight`;
5. запустите сервер;
6. сделайте один тестовый документ.

Смена профиля не должна автоматически менять `archive.root`.
