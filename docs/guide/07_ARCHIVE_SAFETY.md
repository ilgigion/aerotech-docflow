# 7. Защита архива и идемпотентность

## Принцип fail-closed

Если приложение не может доказать, что действие безопасно, оно завершает
операцию ошибкой. Ложная ошибка предпочтительнее ложного успеха и перезаписи
существующего документа.

## Защита до запуска сервера

Production preflight требует:

- `environment = "production"`;
- осмысленную версию, не `dev`;
- localhost host;
- абсолютные пути;
- существующие NAPS2, incoming, logs, idempotency и archive root;
- archive root, не содержащий компонент `archive_test`;
- `archive.confirmation`, точно равный root после resolve;
- marker с совпадающим `archive_id`;
- непустой allowlist типов;
- корректный диапазон лет;
- включённые logging и idempotency;
- служебные каталоги вне архива;
- incoming и archive как отдельные неналоженные деревья.

Если проверка не прошла, production-сервер не открывает порт.

## Защита имени

Финальное имя:

```text
ТИП_ГГММДД_ЧЧММСС_НОМЕР.pdf
```

API отклоняет неоднозначные значения до lock и сканирования. Тип документа
нормализуется в канонический uppercase, а номер должен быть file-safe уже на
входе.

Если базовое имя занято существующим PDF, программа его не трогает и пробует
суффиксы:

```text
..._001.pdf
..._001_01.pdf
..._001_02.pdf
```

Поддерживается до 100 кандидатов. Выбор имени защищён `.reserve`, поэтому два
процесса не могут незаметно выбрать один путь.

## Проверка PDF

После NAPS2 файл должен пройти:

1. существование и тип обычного файла;
2. минимальный размер;
3. заголовок `%PDF-`;
4. завершающий `%%EOF`;
5. строгий разбор `pypdf`;
6. минимум одну страницу;
7. стабильность файла после завершения процесса.

Та же строгая проверка повторяется для копии перед публикацией и для PDF,
возвращаемого из записи идемпотентности.

## Атомарная публикация

```text
incoming source
  → уникальный скрытый .tmp рядом с final
  → flush + fsync
  → сравнение размера
  → SHA-256 source и temp
  → строгая PDF-проверка temp
  → atomic hard link temp → final
  → удаление имени temp
  → удаление source только после успеха
```

Финальный hard link создаётся без overwrite. Если final появился в гонке,
операция завершается ошибкой, а существующий файл не заменяется.

Файловая система архива должна поддерживать hard links; для локального Windows
рекомендуется NTFS. SMB/сетевой архив требует отдельной приёмки на конкретном
сервере.

## Lock сканера

`<INCOMING_DIR>\.scanner.lock` содержит:

- `operation_id`;
- `task_id`;
- PID;
- hostname;
- время создания.

Lock создаётся атомарно. Освобождается только владельцем с совпадающими полями.
Stale takeover допускается только после timeout и если PID-владелец не жив.
Повреждённый или чужой lock не удаляется автоматически.

## Состояния идемпотентности

JSON хранится в:

```text
C:\ProgramData\Aerotech Docflow\data\idempotency
```

Основные состояния:

| Статус | Значение | Повтор |
|---|---|---|
| `processing` | Запрос начат, скан ещё не подтверждён | 409, пока запись не stale |
| `scanned` | В incoming есть проверенный PDF | повторить только storage |
| `storing` | Имя зарезервировано, публикация могла начаться | проверить final и повторить storage безопасно |
| `succeeded` | Final PDF подтверждён | вернуть существующий результат, без скана |
| `failed`/`interrupted`/`timeout` | Операция завершилась ошибкой | возможна новая попытка по правилам записи |

Ключ связан с fingerprint реквизитов. Один ключ с другими `task_id`, типом,
номером или `scanner_profile` получает `idempotency_key_request_conflict`.

## Авария после публикации

Если PDF уже опубликован, но процесс упал до `succeeded`, запись `storing`
содержит точный final path. При повторе программа:

1. проверяет, что final находится внутри archive root;
2. валидирует PDF;
3. при наличии source сравнивает размер и SHA-256;
4. принимает совпадающий final без `_01` и без нового скана.

Если source и final различаются, возвращается
`idempotency_published_file_mismatch`. Оба файла сохраняются для ручного
расследования.

## Когда удаляется временный PDF

Source `PF_*.pdf` удаляется только после успешной публикации и проверки final.
При недоступном архиве или ошибке копирования исходник остаётся для повторного
storage либо ручного восстановления.

## Что система никогда не должна делать

- перезаписывать существующий PDF;
- удалять чужой PDF;
- возвращать `succeeded`, пока публикация не завершена;
- запускать повторный физический скан для подтверждённого `succeeded`;
- доверять пути из JSON за пределами incoming/archive;
- автоматически удалять артефакт с неизвестным владельцем.
