Да, **очередь задач Planfix можно использовать**, и это даже логично для будущей версии. Но важный момент: **очередь Planfix не должна полностью заменять локальный lock сканера**.

Лучшее решение:

```text id="rid8wy"
Planfix = бизнес-очередь задач
local backend = технический исполнитель
local lock = защита физического сканера
```

То есть Planfix может решать вопрос:

```text id="7q1l55"
какую задачу сканировать следующей
```

А локальный lock решает вопрос:

```text id="5mfoc2"
можно ли прямо сейчас занимать физический сканер
```

---

# Почему Planfix-очередь полезна

Если в Planfix есть задачи, статусы и фильтры, можно сделать такой поток:

```text id="y19xwz"
1. Оператор создаёт задачу.
2. Заполняет тип документа, номер, дату.
3. Ставит статус: "Ожидает сканирования".
4. Наш локальный сервис берёт первую задачу из очереди.
5. Ставит статус: "Сканируется".
6. Запускает scanner.py.
7. Переносит файл через storage.py.
8. Записывает имя файла в Planfix.
9. Ставит статус: "Отсканировано".
```

Это хороший подход, потому что Planfix уже становится видимой очередью для сотрудников:

```text id="nwmha6"
что ждёт сканирования
что сейчас сканируется
что завершено
где ошибка
кто создал задачу
какие документы в очереди
```

# Почему Planfix не должен быть единственным lock

Потому что Planfix находится выше по уровню. Он не управляет физическим устройством напрямую.

Проблемы, если использовать только Planfix:

```text id="7ey89m"
1. Два экземпляра backend могут одновременно взять одну задачу.
2. Оператор может два раза нажать "Сканировать".
3. Статус в Planfix может обновиться с задержкой.
4. Может пропасть сеть между backend и Planfix.
5. Задачу могут вручную изменить во время сканирования.
6. Сканер может быть занят NAPS2 GUI или другим процессом.
7. Planfix не знает, что TWAIN-драйвер ещё не освободил устройство.
```

Поэтому даже при Planfix-очереди у нас всё равно должен быть локальный защитный механизм:

```text id="079nz1"
scanner.lock
```

Он говорит:

```text id="4kv8ht"
сканер сейчас занят физически, второй запуск запрещён
```

# Правильная архитектура

Я бы разделил так:

```text id="nuzy6f"
Planfix queue
  управляет задачами и статусами

Local scanner service
  выполняет одну задачу за раз

Local file lock
  защищает физический сканер от параллельного запуска

Local logs / later SQLite
  фиксирует техническую историю операции
```

Схема:

```text id="zcgpwz"
Planfix task
   ↓
local backend получает задачу
   ↓
берёт local scanner lock
   ↓
NAPS2 сканирует
   ↓
PDF переносится в архив
   ↓
Planfix получает имя файла
   ↓
lock освобождается
```

# Практично ли использовать Planfix как очередь?

Да, **практично**, если процесс будет не мгновенной кнопкой, а очередью заданий.

## Хороший вариант

```text id="9svjfu"
Оператор создаёт задачу → задача попадает в очередь → сервис сам забирает и сканирует
```

Плюсы:

```text id="mtye19"
видна очередь
видны статусы
можно отслеживать ошибки
можно повторять задачи
не нужно держать HTTP-запрос открытым 120 секунд
```

## Менее удачный вариант

```text id="nfopae"
Оператор нажал кнопку → Planfix сразу вызывает backend → backend сразу сканирует
```

Так тоже можно, но тогда Planfix работает не как очередь, а как кнопка запуска. В этом варианте lock особенно нужен.

Planfix статусы:

```text id="yprfvg"
Новый
Ожидает сканирования
Сканируется
Отсканировано
Ошибка сканирования
Ошибка сохранения
```

И технические поля:

```text id="a190zw"
operation_id
locked_at
locked_by
scan_started_at
scan_finished_at
file_name
archive_path
error_code
error_message
```

Алгоритм worker:

```text id="9lccer"
1. Найти старейшую задачу со статусом "Ожидает сканирования".
2. Проверить, что она не locked.
3. Присвоить operation_id.
4. Поставить "Сканируется".
5. Взять local scanner lock.
6. Выполнить process_document_scan().
7. Записать file_name.
8. Поставить "Отсканировано".
```

Если ошибка:

```text id="ym00ni"
1. Записать error_code.
2. Записать error_message.
3. Поставить статус "Ошибка сканирования" или "Ошибка сохранения".
4. Освободить local lock.
```
